# Anycast

**BGP over Secure Transport (BGPoST) with Anycast Failover**
Based on: *"The Multiple Benefits of a Secure Transport for BGP"* (Wirtgen et al., CoNEXT 2024)

---

## What This Lab Does

Runs 4 Docker containers simulating a real BGP anycast network where:
- BGP sessions are secured with **TLS encryption + mutual certificate authentication**
- Two replicas share one IP (`192.0.2.1`) and serve DNS
- When one replica's DNS service dies, **BGP automatically withdraws its route**
- Traffic fails over to the healthy replica **without any manual intervention**

---

## Network Layout

```
                     10.10.0.0/24 (docker bridge)

  client-node      provider-router       replica-a          replica-b
  10.10.0.5   ───   10.10.0.2 (AS65000)  10.10.0.3 (AS65010) 10.10.0.4 (AS65011)
                       |                     |                    |
                       |──── BGPoTLS ────────|                    |
                       |──── BGPoTLS ─────────────────────────────|
                       
               forwards 192.0.2.1 traffic
               to whichever replica is healthy
```

## Step 5 — Check BGP Sessions (The TLS-Secured Control Plane)

```bash
sudo docker exec provider-router birdc show protocols
```

Expected — both sessions **Established**:
```
Name       Proto    State  Since         Info
replica_a  BGP      up     17:05:34.348  Established
replica_b  BGP      up     17:05:35.012  Established
```

If you see `Connect` or `Idle`, wait 10-15 seconds and re-run. BGP sessions
take a moment to negotiate the TLS handshake and exchange OPEN messages.

### Check the anycast routes learned via BGPoTLS
```bash
sudo docker exec provider-router birdc show route
```

Expected — `192.0.2.1/32` should appear **twice** (once per replica):
```
192.0.2.1/32   unicast [replica_a 17:05:34.348] * (100) [AS65010i]
               via 10.10.0.3 on eth0
               unicast [replica_b 17:05:35.012]   (100) [AS65011i]
               via 10.10.0.4 on eth0
```

The `*` marks the currently preferred path (replica-a, lower router ID wins).


## Step 6 — Wire Up the Client and Test Normal Operation

### Give client-node a route to the anycast prefix
```bash
sudo docker exec client-node ip route replace 192.0.2.1/32 via 10.10.0.2
```

This tells client-node: "to reach 192.0.2.1, go through provider-router."
This is needed because 192.0.2.1 is not on the docker bridge subnet.

Verify the route exists:
```bash
sudo docker exec client-node ip route show 192.0.2.1
# Expected: 192.0.2.1 via 10.10.0.2 dev eth0
```

### Ping the anycast address
```bash
sudo docker exec client-node ping 192.0.2.1
```

Expected: continuous successful replies. The IP `192.0.2.1` lives on the
loopback of both replicas — provider-router forwards to whichever is preferred.

### Query DNS at the anycast address
```bash
sudo docker exec client-node dig +short TXT identity.bgpost.lab @192.0.2.1
```

Expected:
```
"REPLICA-A"
```

This proves replica-a is answering. Both replicas share the IP but return
different TXT records, so you can see exactly who answered.

## Step 7 — Failover Test (The Main Demo)

### Part A — Start continuous ping in one terminal

Open **Terminal 1**, start ping and leave it running:
```bash
sudo docker exec client-node ping 192.0.2.1
```

### Part B — Simulate failure in another terminal

Open **Terminal 2**, run the failover:

```bash
# Confirm normal state - replica-a answering
sudo docker exec client-node dig +short TXT identity.bgpost.lab @192.0.2.1
# Shows: "REPLICA-A"

# Show both routes present in provider's table
sudo docker exec provider-router birdc show route 192.0.2.1/32

# SIMULATE FAILURE: kill DNS on replica-a
# (container stays up, only the DNS service dies)
sudo docker exec anycast-replica-a pkill -f nsd

# Watch the healthcheck detect it (follow logs)
sudo docker logs -f anycast-replica-a
```

In the logs you will see:
```
[healthcheck] DNS service DOWN on 192.0.2.1! Withdrawing BGP advertisement...
[healthcheck] BGP session restarted - withdrawal in progress.
```

### Part C — Check failover happened

```bash
# Wait for healthcheck cycle (2-4 seconds), then check provider's table
sleep 6
sudo docker exec provider-router birdc show route 192.0.2.1/32
```

Expected — replica_a route is **GONE**:
```
192.0.2.1/32   unicast [replica_b 17:05:35.012] * (100) [AS65011i]
               via 10.10.0.4 on eth0
```

```bash
# Query DNS again - should now return REPLICA-B
sudo docker exec client-node dig +short TXT identity.bgpost.lab @192.0.2.1
# Shows: "REPLICA-B"
```

Meanwhile in Terminal 1 (ping), you should see:
```
64 bytes from 192.0.2.1: seq=10 ttl=64 time=0.12 ms   ← replica-a
64 bytes from 192.0.2.1: seq=11 ttl=64 time=0.11 ms
Request timeout for icmp_seq 12                        ← failover moment
64 bytes from 192.0.2.1: seq=13 ttl=64 time=0.14 ms   ← replica-b
64 bytes from 192.0.2.1: seq=14 ttl=64 time=0.13 ms
```

The 1-2 second gap is the BGP control plane withdrawing and rerouting.
Traffic resumes automatically — the client never changed anything.

### Part D — Check BGP session state
```bash
sudo docker exec provider-router birdc show protocols
```

Expected:
```
replica_a  BGP   start  Idle    Received: Administrative reset
replica_b  BGP   up     Established
```

`Administrative reset` on replica_a confirms the healthcheck script
intentionally restarted the BGP session to force the withdrawal.

## Step 8 — Recovery Test

```bash
# Restore DNS on replica-a
sudo docker exec anycast-replica-a nsd -c /etc/nsd/nsd.conf

# Wait for healthcheck to detect recovery
sleep 10

# Both sessions should be Established again
sudo docker exec provider-router birdc show protocols

# Both routes back in table
sudo docker exec provider-router birdc show route 192.0.2.1/32

# DNS query back to replica-a
sudo docker exec client-node dig +short TXT identity.bgpost.lab @192.0.2.1
# Shows: "REPLICA-A"
```

## Step 9 — Demonstrate the BGPoST Certificate Feature

This shows the simulated X.509 certificate configuration embedding
(from Section 4 / Listing 1 of the paper):

```bash
# Show the JSON config blob embedded in replica-a's certificate
bash scripts/02-extract-cert-profile.sh replica-a
```

Expected output:
```
=== Full certificate text ===
X509v3 Subject Alternative Name:
    DNS:replica-a.bgpost.lab
1.2.3.4.5.6.7.8.1: ...

=== Extracting the embedded JSON config blob ===
Raw extension value:
    {role:client,as:65010,prefixes:[192.0.2.1/32],qos_mbps:100}
```

This JSON is genuinely embedded in the X.509 certificate. In the full
research prototype, the provider's router would parse this automatically
and configure BGP import filters without any manual configuration.

## Quick Reference — All Useful Commands

### Container management
```bash
sudo docker compose up -d                    # start all containers
sudo docker compose down                     # stop all containers
sudo docker compose ps                       # check status
sudo docker compose ps -a                    # check status including stopped
sudo docker compose restart anycast-replica-a  # restart one container
sudo docker compose stop anycast-replica-a     # stop one container
sudo docker compose start anycast-replica-a    # start one container
sudo docker compose build --no-cache           # rebuild all images
```

### BGP session inspection
```bash
sudo docker exec provider-router birdc show protocols          # session states
sudo docker exec provider-router birdc show route              # full route table
sudo docker exec provider-router birdc show route 192.0.2.1/32 # specific route
sudo docker exec provider-router birdc show route all          # routes + attributes
sudo docker exec anycast-replica-a birdc show protocols all provider  # detailed session info
```

### Manual BGP control (for testing)
```bash
sudo docker exec anycast-replica-a birdc disable anycast_route  # withdraw route
sudo docker exec anycast-replica-a birdc enable anycast_route   # re-advertise route
sudo docker exec anycast-replica-a birdc restart provider       # reset BGP session
```

### DNS testing
```bash
sudo docker exec client-node dig +short TXT identity.bgpost.lab @192.0.2.1  # which replica?
sudo docker exec client-node dig +short A service.bgpost.lab @192.0.2.1     # A record
sudo docker exec client-node dig TXT identity.bgpost.lab @192.0.2.1         # full output
```

### Connectivity testing
```bash
sudo docker exec client-node ping 192.0.2.1             # ping anycast IP
sudo docker exec client-node ping 10.10.0.3              # ping replica-a directly
sudo docker exec provider-router ping 10.10.0.3          # provider→replica-a
```

### Log watching
```bash
sudo docker logs provider-router                         # provider logs
sudo docker logs anycast-replica-a                       # replica-a logs
sudo docker logs anycast-replica-b                       # replica-b logs
sudo docker logs -f anycast-replica-a                    # follow live
sudo docker logs --tail 30 anycast-replica-a             # last 30 lines
```

### Kernel routing table (actual packet forwarding)
```bash
sudo docker exec provider-router ip route show           # full kernel table
sudo docker exec provider-router ip route show 192.0.2.1 # specific entry
sudo docker exec anycast-replica-a ip addr show lo       # verify anycast IP on loopback
```

### Packet capture (proves TLS encryption)
```bash
# On provider-router — BGP port traffic should be encrypted bytes
sudo docker exec provider-router tcpdump -i eth0 -n port 179 -X -c 20

# On replica-a — watch DNS traffic
sudo docker exec anycast-replica-a tcpdump -i eth0 -n port 53 -c 10
```
